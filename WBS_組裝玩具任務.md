# 工作項目清單 (WBS) — 組裝玩具任務

> 個人獨立製作,不做人力分配。規模:S(≤1日)/ M(2~3日)/ L(≥4日,警惕)。
> 每項回頭對照核心驗證目標;不服務驗證的標【原型後】。

**核心驗證目標**:用 crank + 方向鍵切換操作「自己組裝出來的機體」是否好玩;目標感受為**手忙腳亂、緊張爽快**。凡不服務於此的一律往後放。

---

## ⚠️ 重要前提:這是「既有原型的重整」,不是從零開發

現有原型**功能已相當完整**:主選單、組裝(HQ)、商店、6 個任務(M001–M006,含 ELIMINATE_ALL 與 DELIVER_STONE 兩型)、7 個零件(三類齊全)、敵人、存檔、音效都在。

因此本 WBS **絕大多數是「修正 / 重構 / 調校」既有系統**,而非新建。已完整的部分(關卡數量、零件、存檔、音效)不再投入。真正要做的,是把**核心操作手感**與**機體整體感**打磨到能驗證假設的程度。

已完成(不列入):初始化死碼修正、根目錄 VSCode build task([PR #2](https://github.com/tppr2046/hayatogam_assamble_toy/pull/2))、自動化編譯+開機 log 驗證流程確認可用。

## 依賴與建議順序(排程前的邏輯)

```
A1 (輸入規格,最先、最便宜)
      │
      ├─→ P2s (斜坡 spike,先驗證方案) ─→ P2 (斜坡正式重構) ─┐
      │        └─(P1 拆檔整備,P2 前完成)                    │
      │                                                      ├─→ A2 (手感調校) ─→ T2 (實玩驗證) ─→ T3 (驗收)
      ├─→ P3 (切換卡卡修正,核心風險) ─→ P4 (切換狀態) ────┘
      │
      └─→ A3 ─→ P5 (組裝 UI 重整) / G1   （與主線平行,低風險）
L1 (選機體與關卡) 可隨時做,越早越好,供 T2 用
```

**風險前置原則**:P2(斜坡)與 P3(卡卡)是兩個決定成敗的高風險項,應盡早撞上。A1 幾乎零成本卻是全部前置,務必第一個完成。P2 之前先以 **P2s spike** 驗證方案,再決定是否展開。
